## 2. 网络方案

### 2.1 现实情况：直面挑战

**Claude.ai、ChatGPT（OpenAI）在中国大陆均无法直接访问。** 这不是可以回避的技术细节——这是日常运营需要解决的核心问题。以下方案基于实际可行性设计。

### 2.2 方案一：机构级VPN/代理（强烈推荐）

**原理**：在学校路由器层面配置透明代理，全教室设备自动走代理，学生无需手动操作。

**实施步骤**：
1. 购置支持OpenWrt的软路由（如友善NanoPi R4S，约¥500）
2. 购买企业级机场服务或专线（如Shadowsocks/Trojan协议，参考价¥200–500/月）
3. 在软路由配置分流规则：国内流量直连，AI工具流量走代理
4. 测试稳定性：连接Claude.ai、OpenAI API端点，延迟应<300ms

**优点**：
- 学生透明使用，无感知
- 集中管理，一处故障统一排查
- 可配置白名单，仅开放必要域名

**风险与缓解**：
- 代理服务商可能被封锁→ 同时保留2–3个备用服务商
- 政策合规风险→ 参考学校法律顾问意见，仅用于教育目的

### 2.3 方案二：学生个人设备代理（备用）

当学校代理出现故障时，可临时要求学生使用手机热点+个人VPN。适合应急，不适合日常。

### 2.4 方案三：使用国内可访问的替代工具（备用+补充）

部分课堂活动可切换至国内可直接访问的AI工具（见工具对比矩阵）。这是网络中断时的二级应急方案，不是主方案。

### 2.5 网络架构示意

```
互联网
  ↓
软路由（OpenWrt + 透明代理）
  ↓ 分流
  ├─ 国内域名 → 直连宽带
  └─ Claude/OpenAI/GitHub → 代理节点（海外）
  ↓
24口千兆交换机
  ↓
15台学生工作站 + 教师机
```

### 2.6 网络速度要求估算

| 场景 | 单台带宽需求 | 15台并发需求 |
|------|-------------|-------------|
| 使用Claude网页版聊天 | 1–2 Mbps | 15–30 Mbps |
| ClaudeCode/API调用 | 2–5 Mbps | 30–75 Mbps |
| 代码托管（GitHub推送）| 0.5 Mbps | 7.5 Mbps |
| **建议宽带套餐** | — | **300 Mbps 对称** |

## 3. AI工具账号管理方案

### 3.1 分阶段账号策略

| 课程阶段 | 工具 | 账号模式 | 理由 |
|----------|------|----------|------|
| **对话者** | Claude.ai 网页版 | 共享 Pro 账号（轮用）或每人 Free | 入门阶段，用量小 |
| **创造者** | Claude.ai + Cursor | 机构 Team 账号 / Cursor Pro | 需要稳定使用量 |
| **建造者** | Claude Code / OpenCode | 机构 API Key（统一管理）| 代码工具必须用 API |
| **设计者** | 全套工具 | 机构 API Key + Team 账号 | 高级阶段，自主性强 |

### 3.2 API Key 管理方案（建造者/设计者阶段核心）

**架构原则**：一个机构主 API Key，通过中间层分发给学生，避免直接暴露原始密钥。

```
Anthropic 官方账号
  ↓
机构主 API Key（教师掌管，从不共享）
  ↓ 通过 Anthropic Console 子密钥 或 开源代理层
  ↓
每个学生：独立的受限密钥
  - 每日 Token 上限：200,000 tokens
  - 每分钟请求限制：10 次
  - 可用模型：Haiku 4.5（日常）/ Sonnet 4.6（特殊任务，需教师授权）
```

**工具选择**：
- **Anthropic Console**：原生支持 API Key 管理，可设置每月用量上限（强烈推荐）
- **LiteLLM Proxy**（开源）：可在本地服务器架设代理层，统一分配密钥，监控用量

### 3.3 账号安全规范

1. **密钥存储**：所有 API Key 存于教师专用密码管理器（Bitwarden）
2. **学生密钥**：通过 `.env` 文件配置，禁止写入代码文件并上传 GitHub
3. **轮换周期**：每学期更换一次主密钥
4. **离职处理**：TA 离职后48小时内撤销其访问权限

### 3.4 订阅账号分配

| 账号类型 | 数量 | 用途 |
|----------|------|------|
| Claude Team（标准席位）| 5–10 个 | 创造者阶段学生轮流使用 |
| Cursor Pro | 15 个 | 建造者阶段每人一个 |
| GitHub Free | 15 个 | 学生代码托管 |
| Anthropic API 账号 | 1 个（机构）| 统一管理所有 API 费用 |

### 4.3 API 费用计算（月度）

#### Haiku 4.5（主力模型，80%用量）

```
每月对话总次数 = 15人 × 30次/天 × 20天 = 9,000次
Input tokens/月  = 9,000次 × 500 tokens × 80% = 3,600,000 tokens = 3.6M tokens
Output tokens/月 = 9,000次 × 300 tokens × 80% = 2,160,000 tokens = 2.16M tokens

Input费用  = 3.6M × $0.80/M  = $2.88
Output费用 = 2.16M × $4.00/M = $8.64
Haiku小计  = $11.52/月
```

#### Sonnet 4.6（补充模型，20%用量）

```
Input tokens/月  = 9,000次 × 500 tokens × 20% = 900,000 tokens = 0.9M tokens
Output tokens/月 = 9,000次 × 300 tokens × 20% = 540,000 tokens = 0.54M tokens

Input费用  = 0.9M × $3.00/M  = $2.70
Output费用 = 0.54M × $15.00/M = $8.10
Sonnet小计 = $10.80/月
```

## 6. API费用失控防护机制

### 6.1 三层防护架构

```
层1：Anthropic Console 账户层预算
  ↓ 超出→ 整个账户停止调用
层2：学生密钥层每日限额
  ↓ 超出→ 该学生今日停止调用
层3：代码层输入Token长度限制
  ↓ 防止意外上传超长文件
```

### 6.2 Anthropic Console 账户层设置

1. **月度硬性预算上限**：在 Console → Billing → Usage limits 设置
   - 建议设置：$100/月（约为预期用量的3倍）
   - 触达80%时：邮件预警给教师
   - 触达100%时：自动停止

2. **API Key 粒度控制**（Anthropic Console 支持按 Key 设置限制）：
   ```
   每个学生密钥配置：
   - 日用量上限：$2/天（约对应60,000 Haiku tokens）
   - 周用量上限：$8/周
   ```

### 6.3 课堂操作规范

| 规则 | 内容 |
|------|------|
| 禁止上传大文件 | 严禁将>1000行的文件直接粘贴进prompt（先请教TA如何分段）|
| 循环调用警告 | 代码中的API循环调用必须先估算Token消耗，并经教师审批 |
| 每日使用查看 | 每节课结束时，教师打开 Console 查看当日用量 |
| 异常提醒 | 发现单日消耗超过学生均值3倍时，与该学生确认是否误操作 |

### 6.4 速率限制配置（LiteLLM Proxy 方案）

若使用 LiteLLM 作为代理层，可精细化配置：

```yaml
# litellm_config.yaml 示例
model_list:
  - model_name: haiku-student
    litellm_params:
      model: claude-haiku-4-5
      api_key: os.environ/ANTHROPIC_KEY

router_settings:
  num_retries: 2
  timeout: 30

general_settings:
  max_budget: 100        # 月度总预算 $100
  budget_duration: 1mo

# 每学生限额（通过virtual key实现）
# student_key_budget: $2/day
```

### 6.5 异常费用响应流程

```
发现费用异常（超预警值80%）
  ↓
教师登录 Anthropic Console → 查看 Usage Analytics
  ↓ 判断原因
  ├─ 正常用量高峰 → 调整预算上限
  ├─ 特定学生异常 → 临时禁用该学生密钥，当面确认
  └─ 密钥泄露 → 立即吊销全部密钥，重新生成，通知Anthropic安全
```

## 7. 网络与工具中断应急方案

> **原则**：每节课备有至少一个"离线版本"，教师永远不应该对网络中断束手无策。

### 7.1 应急响应等级

| 中断类型 | 应急响应 |
|----------|----------|
| 代理短暂故障（<30分钟）| 切换备用代理节点，继续上课 |
| 代理当日无法使用 | 切换至国内AI工具（DeepSeek/通义）|
| 全部AI工具不可用 | 执行离线活动方案（见7.2）|
| 停电/硬件故障 | 执行纸笔+口头活动方案（见7.2 活动3）|

### 7.2 离线备用活动（三个具体方案）

---

#### 🎮 离线活动一：「算法大厨」角色扮演

**适合年龄**：6–12岁  
**时长**：45–60分钟  
**所需材料**：
- 每人一张"食谱任务卡"（预先打印，每张描述一道菜：如"做番茄炒蛋"）
- 便利贴（黄色=步骤，红色=判断，绿色=开始/结束）
- 白板或大张白纸

**步骤**：
1. **（5分钟）引入**：教师说："今天AI罢工了，我们亲自体验AI的思维方式——算法！一个程序就像一张做饭的菜谱，必须精确到每一步。"
2. **（10分钟）示范**：教师用便利贴在白板上贴出"泡方便面"的流程图，包含判断（"水开了吗？→是/否"）
3. **（25分钟）独立创作**：每人（或两人一组）拿到任务卡，用便利贴在白纸上创作自己的"算法流程图"。要求：必须有至少1个判断节点（菱形）
4. **（15分钟）互相"执行"**：交换流程图，对方严格按照流程图说的做（口头模拟），找出流程图中的BUG（逻辑漏洞）
5. **（5分钟）总结**：讨论"为什么AI需要我们把指令说得这么精确？"

**学习目标**：体验算法思维，理解为什么给AI的prompt需要精确

#### 📖 离线活动二：「提示词诊所」纸上分析

**适合年龄**：10–15岁  
**时长**：45分钟  
**所需材料**：
- 预先打印的"坏prompt样本卡"（10张，每张一个糟糕的提示词例子）
- 评分表格（打印版）
- 彩色笔

**样本卡内容示例**：
```
卡片1（糟糕）："帮我写代码"
卡片2（糟糕）："解释一下Python"
卡片3（中等）："写一个计算1+1的Python程序"
卡片4（较好）："用Python写一个函数，输入两个整数a和b，返回它们的和，并在函数下方写一个测试例子"
```

**步骤**：
1. **（5分钟）发放材料**，说明规则
2. **（20分钟）诊断阶段**：每人独立对每张卡片打分（1–5分），并用红色笔标出"缺少的关键信息"，用绿色笔写出"如何改进"
3. **（15分钟）讨论**：小组分享诊断结果，教师引导讨论"什么让一个prompt变好？"（角色、上下文、具体要求、输出格式）
4. **（5分钟）总结**：每人写下"我的提示词3原则"

**学习目标**：建立提示词思维，即使没有AI也能思考如何与AI更好沟通

#### 🖊️ 离线活动三：「低科技游戏设计」纸笔原型

**适合年龄**：8–15岁  
**时长**：60–90分钟  
**所需材料**：
- A4纸若干
- 彩色笔/铅笔
- 剪刀（可选）
- 游戏设计工作表（打印版，含：游戏名称、玩家人数、核心规则3条、获胜条件）

**步骤**：
1. **（10分钟）回顾**：教师说"上次我们用AI帮我们设计了一个游戏，今天我们自己来——这是AI背后人类创意的部分！"回顾游戏设计的关键要素（目标、规则、反馈、挑战）
2. **（5分钟）分组**：2–3人一组，抽签决定游戏主题（如"农场""太空""厨房"）
3. **（35分钟）设计**：
   - 用游戏设计工作表完成设计文档
   - 制作纸质游戏原型（卡牌/棋盘/道具）
   - 自己先试玩一遍，找出规则漏洞
4. **（20分钟）互相试玩**：组间交换，每组试玩另一组的游戏，并填写反馈表
5. **（10分钟）反思**：讨论"如果现在让你给AI出一道'设计这个游戏'的指令，你会怎么说？"

**学习目标**：理解游戏设计的人类创意部分，体验产品迭代；为之后的AI辅助设计打下认知基础

## 8. 远程兼职TA方案

### 8.1 角色定义：AI沟通引导师

**TA的核心职责是帮助学生更好地与AI沟通，而不是替学生写代码。**

这一区别至关重要：
- ✅ TA做的："你试着把你的问题说得更具体一些，告诉Claude你已经试过什么了"
- ✅ TA做的："你看Claude给的错误信息，它说的是第3行——你觉得第3行哪里有问题？"
- ✅ TA做的："你的prompt缺少了上下文，加上你在做什么项目，再问一次试试"
- ❌ TA不做的：直接给出完整解决方案代码
- ❌ TA不做的：接管学生电脑远程操控

### 8.2 TA工作模式

| 维度 | 详情 |
|------|------|
| **工作类型** | 远程兼职，按小时计费 |
| **沟通工具** | 飞书/微信工作群 + 视频通话（腾讯会议/Zoom）|
| **异步支持** | 学生在群里发问题截图，TA在指定时间窗口内回复 |
| **同步支持** | 每周2次在线答疑（各30分钟）|
| **响应时间要求** | 上课时间内：≤15分钟；课外：≤2小时 |

### 8.3 TA工作量估算

| 工作内容 | 时间/周 |
|----------|---------|
| 异步问题回复（群消息）| 3–5小时 |
| 在线同步答疑（每次30分钟×2）| 1小时 |
| 课程材料辅助准备（可选）| 1–2小时 |
| **每周总工时** | **5–8小时** |
| **每月总工时** | **20–32小时** |

### 8.4 TA角色要求

| 要求项 | 最低标准 |
|--------|----------|
| 技术背景 | 能独立用Claude完成简单编程项目（个人作品可作证明）|
| 沟通方式 | 善于用问题引导，而非直接给答案（Socratic方法）|
| 年龄适应 | 有与6–15岁学生互动经验（家教/辅导经历均可）|
| 时间稳定 | 每周承诺固定5–8小时，持续至少一学期 |
| 中文水平 | 母语中文，可用中文解释技术概念 |

### 8.5 TA成本估算

| 场景 | 时薪 | 月工时 | 月成本 |
|------|------|--------|--------|
| 在校大学生（CS专业）| ¥50–80/时 | 20–32小时 | ¥1,000–2,560 |
| 初级程序员/应届毕业生 | ¥80–120/时 | 20–32小时 | ¥1,600–3,840 |
| **预算基准** | ¥80/时 | 25小时 | **¥2,000/月** |

> **TA成本已纳入月度总成本计算（第4节）**

### 8.6 TA招募渠道

1. **高校CS/计算机专业** — 发布校园招聘或联系导师推荐
2. **GitHub/开源社区** — 寻找有教育热情的开发者
3. **AI学习社区（如少楠Planet、即刻AI圈）** — 找对儿童教育有兴趣的AI爱好者
4. **前学生家长网络** — 有技术背景且认同教育理念的家长

### 8.7 TA标准问题引导卡（参考）

TA在回答学生问题时优先使用以下引导方式：

```
"你自己觉得问题出在哪里？"
"你已经试过什么方法了？"
"Claude给的错误信息是什么？"
"如果你要向一个不懂代码的人解释你想做什么，你会怎么说？"
"这个问题有没有可以拆成更小的部分？"
"试着把你的问题重新问Claude，加上你现有的代码和你已经试过的"
```

## 10. 屏幕时间管理策略

### 10.1 核心原则

屏幕时间管理目标不是减少使用时长，而是**保证使用质量、防止依赖、保护身体健康**。

### 10.2 每日课堂时间结构

| 时段 | 内容 | 屏幕状态 |
|------|------|----------|
| 第1–5分钟 | 签到、目标说明 | 关闭/待机 |
| 第6–25分钟 | 主题讲解+小组讨论 | 关闭（聚焦听讲）|
| 第26–55分钟 | 动手实践（使用AI/写代码）| 开启（聚焦工作）|
| 第56–60分钟 | 反思总结、收尾 | 关闭（口头分享）|
| 每45分钟后 | 强制5分钟休息 | 关闭 |

> **20-20-20护眼规则**（嵌入课堂常规）：每20分钟，看20英尺（6米）外的东西20秒。教师设置提醒。

### 10.3 屏幕行为规范

| 规则 | 内容 |
|------|------|
| **单任务原则** | 工作时间只开与任务相关的标签页，其他全部关闭 |
| **主动提问先于AI查询** | 遇到问题先思考1分钟，写下自己的想法，再去问AI |
| **禁止无目的浏览** | 课堂时间不得访问与课程无关的网站/游戏 |
| **分享优于独占** | 发现有趣的AI回复，鼓励截图分享给全班 |

### 10.4 家长沟通与家庭协议

- 每月向家长发送《本月课程技术使用报告》（学生用了什么工具、做了什么项目）
- 建议家长设置家庭网络AI使用时段（如晚8点后不得使用AI工具做作业）
- 课程不鼓励学生在家"自学"AI工具（避免无引导的过度使用），但可以展示课堂成果

### 10.5 AI依赖风险与干预

**识别信号**：
- 学生遇到任何小问题立即问AI，不再尝试独立思考
- 无法在无AI情况下完成基础任务（如离线活动时明显焦虑）
- 对AI回复不加批判地全盘接受

**干预措施**：
1. 每两周安排一节"无AI挑战课"（完成类似任务，但全程不用AI）
2. 引导学生维护"我的学习日志"，记录"今天我自己想到了什么"
3. 定期回顾：让学生展示"是AI帮我做的"vs"是我自己想的"的比例
