# 阶段四课程：设计者（Designer）

**本妙坊未来成长中心 · AI创造课 · L4 顶石阶段**

---

## 阶段概览

### 这一阶段在做什么

阶段四是整个课程的终点，也是真正意义上的起点。

在前三个阶段，学生学会了跟AI说话、从零构建作品、分模块建造应用。到了阶段四，题目不再由老师给。问题从哪来？从生活里来。解决谁的问题？从用户调研里找。做出来之后怎么知道好不好？用真实用户的反应来判断。

这一阶段培养的不是"会用AI的人"，而是**会发现问题、有能力用AI解决问题的人**。

### 入口条件

进入阶段四，学生需要完成阶段三出口评估：

- 能在动手前画出或描述整个作品的结构
- 能分步骤完成多模块应用，并自己测试每个部分
- 收集真实用户反馈后，能根据反馈修改作品
- 独立完成一个被真实用户实际使用过的作品

### 出口条件

阶段四结束（Demo Day 展示后），学生需能独立完成：

- [ ] 识别出一个真实存在的问题，并说清楚"谁有这个问题，有多频繁，解决了有什么价值"
- [ ] 独立规划整个项目的推进步骤，并按计划执行
- [ ] 作品经历至少两轮真实用户测试与修改
- [ ] 向公开观众（家长、同学）展示并说明作品的设计决策

### 核心能力

| 能力 | 具体说明 |
|------|---------|
| **问题发现** | 不依赖老师给题目，主动在生活中发现值得解决的问题 |
| **需求验证** | 通过询问潜在用户，确认问题是否真实、是否值得解决 |
| **独立规划** | 从想法到完成，自己制定时间线和步骤 |
| **迭代设计** | 经历至少两轮真实测试，每次基于反馈改进 |
| **公开表达** | 在 Demo Day 上向家长和同学清晰展示作品和设计决策 |

### 阶段时长

**建议：16-24 课时**（不含 Demo Day 和准备课时）

- 问题发现工作坊：3-4 课时
- 需求分析与项目规划：2-3 课时
- 独立构建与迭代：8-12 课时
- Demo Day 准备与展示：3-4 课时

### 工具清单

进入阶段四时，学生已熟悉以下工具，不需要引入新工具：

| 工具 | 用途 |
|------|------|
| ClaudeCode / OpenCode | 主要构建环境，与 AI 协作构建应用 |
| 截图工具 | 记录过程、向 AI 描述问题 |
| 纸笔 | 问题地图、用户旅程图、结构草图 |

阶段四最多引入 **1 个新工具**（可选）：简单的在线表单工具（如腾讯问卷），用于收集用户调研。如果中心没有条件，纸质问卷可以替代。

## 能力迁移说明

### 辩论 → 需求收集

辩论课已经训练了学生一种能力：为一个立场找论据，同时理解对方立场的逻辑。这种能力在阶段四有一个直接的映射。

| 辩论中训练的 | 在需求收集中变成了 |
|------------|----------------|
| **立论**：说清楚"为什么我的立场是对的" | **问题定义**：说清楚"为什么这个问题值得解决，对谁有价值" |
| **驳论**：找到对方论点的漏洞，说"你的这个前提不成立" | **需求筛选**：对用户提出的需求说"这个需求是真实痛点，还是一时想到的"，区分"想要"和"真正需要" |
| **提问环节**：向对方提出精准问题，挖出他们没说清楚的假设 | **用户访谈**：问用户"你现在怎么解决这个问题""你多久遇到一次"，而不是直接问"你要不要用这个功能" |
| **四辩总结**：在有限时间内，把整场辩论的最核心论点说清楚 | **Demo 展示**：在3分钟内，说清楚"问题是什么、给谁解决、我怎么解决的" |

**实操提示：** 做用户访谈时，可以把辩题改造成访谈框架。把"我的立场是XXX"换成"我的作品解决的是XXX问题"，把"反驳"换成"用户说'用不上'时，你怎么回应这个质疑"。

**辩论教师联动：** 在辩论课上，可以让学生选一个同学的 AI 项目作为"己方立场"，辩护"这个项目值得被做"。这等于做了一次需求分析练习。

---

### 哲思 → 伦理设计

哲学思辨课训练了一种能力：在有争议的问题上，不急着下结论，而是先把所有相关的价值观摆出来，理解张力，再做判断。这个能力在设计 AI 应用时特别有用。

| 哲思中训练的 | 在伦理设计中变成了 |
|------------|----------------|
| **苏格拉底式追问**："你说'公平'，公平到底是什么意思？" | **需求深挖**："用户说'要方便'，方便对他来说到底是什么？是少按几下，还是不用记密码，还是别的？" |
| **立场悬置**：暂时放下自己的观点，认真理解对立观点 | **用户同理心**："我做这个工具是为了某类用户，但有没有另一类用户会因此受到影响？" |
| **价值冲突分析**：两个都有道理的原则之间怎么取舍 | **设计取舍**："收集用户数据可以改进产品，但用户有没有权利不被收集？这两个都有道理，怎么选？" |
| **伦理判断**：区分"技术上能做"和"应不应该做" | **AI 伦理意识**："这个功能技术上可以做，但做了对用户公平吗？有没有人会被伤害？" |

**实操提示：** 每次完成一个新功能后，可以做一个30秒的"伦理快检"：这个功能会不会被滥用？收集了什么数据？用户知道吗？不需要很长，养成习惯就够了。

**哲思教师联动：** 设计"AI 作品伦理法庭"——学生展示作品，其他同学轮流扮演"受影响的不同用户"（比如：隐私很重视的用户、年龄很小的用户、不懂技术的老人），提出伦理质疑，设计者需要回应。

---

### 围棋 → 系统规划

阶段三已经用围棋思维训练了结构化规划。到了阶段四，项目更长、不确定性更高，围棋的全局思维有了更深的用武之地。

| 围棋中训练的 | 在顶石项目中变成了 |
|------------|----------------|
| **布局**：在棋盘上规划势力范围，哪里先走、哪里以后再说 | **项目立项**：确定核心功能（最重要的那个先做），其余的"先搁着，等基础做好再加" |
| **中盘**：对手走了意外的棋，计划要调整，但目标不变 | **迭代调整**：用户测试带来了意外发现，原来的设计要改，但核心问题不变 |
| **收官**：优势已定，把每一个角落都收干净，不给对方机会 | **打磨完善**：功能都做完了，逐一测试边缘情况，把"能用"变成"好用" |
| **不抢占所有地盘**：好棋手不贪，知道什么时候该放弃某块 | **功能裁剪**：好的设计不是功能越多越好，知道什么功能不做，作品反而更好 |
| **复盘**：输了也好赢了也好，坐下来把每一步的对错分析清楚 | **项目回顾**：Demo Day 后，回顾整个项目，哪个决策带来了最大的麻烦，下次怎么做 |

---

## 课程结构总览

| 课时 | 主题 | 核心能力 |
|------|------|---------|
| 第1课 | 问题发现工作坊 | 从生活中识别真实问题 |
| 第2课 | 用户调研与需求分析 | 验证问题，找到真实需求 |
| 第3课 | 设计决策与项目规划 | 从需求到方案，制定执行计划 |
| 第4课 | 同伴评审与迭代 | 给出和接受有质量的反馈 |
| 顶石项目 | 独立构建与展示 | 综合运用 L4 所有能力 |
| Demo Day | 成果展示日 | 公开展示，向真实观众说清楚 |

**估计总课时：16-24 课时**（不含 Demo Day）

## 课时一：问题发现工作坊

### 小学版（6-10岁）

**🎯 教学目标**

学生能从自己的日常生活中，主动发现至少一个让人觉得"不方便"或"不舒服"的真实情况，并用一句话说清楚"谁遇到了这个问题，什么时候遇到"。

**⏱ 时间安排**（50分钟）
- 导入（10分钟）：用故事引入"发现问题"的概念
- 主活动（30分钟）：问题地图工作坊
- 分享（10分钟）：每人展示自己发现的一个问题

**📦 所需材料**
- A3 白纸，每人一张（用来画问题地图）
- 彩色便利贴（两种颜色）
- 彩色记号笔
- 教师准备的"问题故事"卡片（见下方流程说明）

**📋 活动流程**

1. **导入故事（10分钟）**

   老师讲一个真实的小故事，例如："有个8岁的小朋友叫小美，她发现每次吃午饭，总有一道菜是她不喜欢的，但她不知道今天有什么菜，只能每次走到餐厅才发现。她想——要是能提前知道今天有什么菜就好了。"

   问学生：这是一个真实的问题吗？谁会遇到这个问题？怎么知道有多少人有这个问题？

2. **问题地图引导（5分钟）**

   老师在黑板上画一个简单的"问题地图"示意：
   - 中间写"我一天的生活"
   - 向外延伸：早上、上学路上、课间、午饭、下午、放学、晚上

   告诉学生：每一个时间段里，有没有哪个时刻让你觉得"这件事真麻烦"或者"要是有个工具帮我做这个就好了"？

3. **独立画问题地图（20分钟）**

   每位学生拿出 A3 纸，在中间写上"我的一天"，按时间段向外延伸，把今天以及最近几天遇到过的"小麻烦"、"不方便"、"重复做的烦事"都写下来。

   便利贴用法：
   - 黄色便利贴 = 我自己觉得麻烦的事
   - 粉色便利贴 = 我观察到别人（家人/同学）觉得麻烦的事

   老师巡视时引导的问题：
   - "这件事你多久遇到一次？"
   - "只有你一个人觉得麻烦，还是其他同学也有这个感受？"
   - "如果有个工具帮你做这件事，你愿意用吗？"

4. **分享与讨论（10分钟）**

   邀请3-4位学生来分享自己发现的一个问题，用这个格式说：
   - "我发现的问题是……"
   - "这个问题发生在……（什么时候）"
   - "遇到这个问题的人有……（谁）"

   班级其他同学举手反应：你也有这个问题吗？

5. **选一个问题（贯穿整个阶段四）**

   最后5分钟，学生从自己的问题地图上，圈出一个"我最想解决的问题"，这将成为他们顶石项目的出发点。

**🎨 预期学生产出**

- 一张填满便利贴的"问题地图"（不少于5个问题点）
- 一个被学生圈出的"主问题"——用一句话写清楚（格式："当[谁]在做[什么事]时，会遇到[什么不方便]"）

示例好产出：

> "当我放学回家想做作业时，总记不清楚老师说了哪些要做，要来回翻书包找。"

示例差产出（太模糊，需要引导改进）：

> "我觉得学习很无聊。"

**📊 评估标准**

| 维度 | 未达到 | 进步中 | 已掌握 |
|------|--------|--------|--------|
| **问题真实性** | 问题很模糊（"生活不方便"），无法说明谁受影响 | 说出了一个具体场景，但还说不清楚是谁遇到的 | 能说出具体的人、具体的时间、具体的不方便，问题可被验证 |
| **问题数量与多样性** | 便利贴少于3张，集中在一个场景 | 有4-5个问题，但都很类似 | 有5个以上来自不同时间段、不同对象的问题 |
| **主动性** | 需要老师帮提示才写出问题 | 自己写了问题，但停下来等老师引导 | 主动在地图上写，不需要老师催，甚至发现"别人的问题" |

**🌐 离线备用活动**

完全不需要电脑。本活动是纸笔工作坊，离线就是标准状态。

如果连便利贴也没有，可以直接用白纸画格子，每个格子写一个问题。

---

#### 中学版（11-15岁）

**🎯 教学目标**

学生能走完"观察→提炼→验证"的问题发现流程，发现一个可以通过初步询问确认其存在的真实问题，并用"问题陈述"格式写出来。

**⏱ 时间安排**（50分钟）
- 导入（10分钟）：分析一个真实应用"是怎么发现问题的"
- 主活动（30分钟）：问题发现工作坊 + 初步验证
- 分享（10分钟）：问题陈述互相评审

**📦 所需材料**
- A3 白纸或素描本
- 便利贴两色
- 笔
- "问题陈述"写法参考卡（见下方）

**📋 活动流程**

1. **引入：产品背后的问题（10分钟）**

   老师展示一个学生熟悉的应用（比如班级通知群、食堂菜单、借书记录表），问学生：

   "这个东西是为了解决什么问题才被做出来的？如果没有它，会发生什么？"

   引导学生理解：每个存在的工具，背后都有一个"没有这个工具时的不方便"。发现了那个不方便，就找到了做工具的理由。

2. **问题发现：三个视角（15分钟）**

   学生在纸上分三栏：

   **第一栏：我自己的摩擦点**
   在过去一周，你有没有做过某件事时觉得"这太麻烦了"？或者"这件事我做了好多遍"？写下来。

   **第二栏：我身边人的麻烦**
   你观察过家人、同学有什么事情做起来很费力吗？有没有看到他们在重复做一件本来可以用工具替代的事？

   **第三栏：缺少的信息或连接**
   有没有什么信息是你需要但找不到的？有没有什么"两个人需要协调但很麻烦"的情况？

3. **问题陈述（10分钟）**

   学生选出自己最感兴趣的一个问题，用以下格式写出"问题陈述"：

   ```
   我发现，当[具体用户群体]在[具体场景/时间]时，
   需要[做某件事]，
   但现在[没有工具/现有工具不好用/只能靠记忆/需要别人协助]，
   导致[具体的不方便/损失/浪费]。
   ```

   示例好的问题陈述：

   > 我发现，当班里同学下午快放学时，需要确认今天作业有哪些，但现在只能靠在黑板上拍照，有时候照不清楚，有时候忘了拍，导致晚上不确定做了没有，要发消息问同学。

4. **两人互审问题陈述（5分钟）**

   两人交换问题陈述，问对方：
   - "这里的'用户'是谁？你说的具体吗？"
   - "'不方便'是真的不方便，还是你觉得不方便？你问过其他人吗？"

**🎨 预期学生产出**

- 一张三栏式问题发现工作纸（每栏至少2个问题点）
- 一份格式完整的"问题陈述"（符合上述模板）
- 问题陈述上的同伴评审批注（至少2条）

**📊 评估标准**

| 维度 | 未达到 | 进步中 | 已掌握 |
|------|--------|--------|--------|
| **问题具体性** | 问题陈述笼统（"生活不方便"），无法识别具体用户和场景 | 有用户和场景，但不方便的程度说不清 | 问题陈述完整填写四个要素，可以被他人理解和验证 |
| **多视角发现** | 只写了自己遇到的问题 | 两栏有内容，有观察他人的尝试 | 三栏都有内容，且来自不同观察视角 |
| **批判性思考** | 同伴评审时只写"很好" | 能指出问题陈述中模糊的地方 | 能提出具体问题，引导对方更精准地描述，或指出"这个可能不是真问题"的可能 |

**🌐 离线备用活动**

本活动完全是纸笔工作，不需要任何电子设备。

额外挑战（有余力的学生）：走到其他同学或学校工作人员旁边，观察他们在做什么，回来写"我观察到的第三栏问题"，不允许直接问对方"你有什么麻烦"，只靠观察。

## 课时二：用户调研与需求分析

### 小学版（6-10岁）

**🎯 教学目标**

学生能向至少2位真实用户提问，验证自己在课时一发现的问题是否真实存在，并收集到至少一条"我没想到"的信息。

**⏱ 时间安排**（50分钟）
- 导入（10分钟）：区分"我猜"和"我知道"
- 主活动（30分钟）：设计问题 + 实际访谈练习
- 整理与分享（10分钟）：记录发现，班级分享

**📦 所需材料**
- 用户访谈记录表（每人2张，见下方模板）
- 笔
- 上节课的问题地图和"主问题"

**用户访谈记录表模板：**

```
我要问的人：______（关系，不用写名字，如"同班同学""妈妈"）
我要验证的问题是：____________________________________

访谈前我猜他会说的：___________________________________

实际问了3个问题：
问题1："你有没有遇到过[某种情况]？"
对方的回答：__________________________________________

问题2："这种情况多久发生一次？"
对方的回答：__________________________________________

问题3："你现在怎么处理这个问题？"
对方的回答：__________________________________________

最让我意外的回答：_____________________________________
我改变了什么想法：_____________________________________
```

**📋 活动流程**

1. **"我猜"和"我知道"的区别（10分钟）**

   老师举一个例子：学生做了一个"帮忙记作业"的应用，但没有问过任何同学，就假设"大家都想要"。

   问班级：这是"我猜大家想要"还是"我知道大家想要"？区别在哪里？

   引出：在做任何应用之前，先去问真实的人，把"我猜"变成"我知道"。这就是用户调研。

2. **设计3个问题（10分钟）**

   学生为自己上节课选的"主问题"，设计3个访谈问题。

   老师示范好问题和差问题的区别：

   | 差的问题 | 好的问题 |
   |---------|---------|
   | "你想用我的应用吗？"（答案几乎总是"想"） | "你最近一周有没有遇到XXX情况？" |
   | "你觉得记作业难不难？" | "你用什么方法记作业？上次忘了作业是什么时候？" |

   原则：**问他们的经历，不要问他们的意见。**

3. **实际访谈练习（15分钟）**

   课堂内两人配对，互相访谈。
   - 一人扮演"设计者"（提问）
   - 一人扮演"真实用户"（回答自己的真实感受，不要配合对方）

   "设计者"填写访谈记录表。

4. **课外访谈任务布置**

   这节课结束后，学生需要在当天或第二天，找到至少1位课堂以外的真实用户（家人、其他班同学）进行访谈，填写第2张记录表。下节课带来。

5. **分享（5分钟）**

   请2-3位学生分享：访谈中最让他们意外的一个回答是什么？

**🎨 预期学生产出**

- 课堂内完成的1张访谈记录表
- 课外完成的1张访谈记录表（下节课带来）
- 一句话总结："通过访谈，我发现我原来以为……，但实际上……"

**📊 评估标准**

| 维度 | 未达到 | 进步中 | 已掌握 |
|------|--------|--------|--------|
| **问题质量** | 问题导向性强（"你要不要用这个"），无法得到真实信息 | 问了行为类问题，但有1-2个导向性问题混入 | 3个问题都是行为类，能挖出真实使用习惯而非意见 |
| **记录完整性** | 记录表填写不完整，对方说的话没有记下来 | 记录了对方的回答，但很简单（一两个字） | 记录了对方的原话（或近似原话），包括"意外发现"部分 |
| **反思** | 没有写改变了什么想法 | 写了改变的想法，但只是重复了对方说的话 | 能指出"我原来以为X，但访谈后发现Y"，并说明这对项目的影响 |

**🌐 离线备用活动**

不需要任何电子设备。

如果当节课无法找到真实用户访谈，可以做"角色扮演访谈"：老师扮演一个典型用户，全班一起向"这个用户"提问，老师根据这类用户的真实情况回答。

---

#### 中学版（11-15岁）

**🎯 教学目标**

学生能设计并执行一次5人以上的用户调研，区分"用户表达的需求"和"用户背后的真实需求"，并产出一份"需求分析摘要"。

**⏱ 时间安排**（50分钟）
- 导入（10分钟）：用真实案例理解"问题背后的问题"
- 主活动（30分钟）：访谈设计 + 角色扮演练习
- 需求分析（10分钟）：从访谈数据提炼真实需求

**📦 所需材料**
- 访谈记录表（中学版，见下方）
- 需求分析工作纸

**📋 活动流程**

1. **"表面需求"和"真实需求"（10分钟）**

   老师讲经典案例：有人说要一匹更快的马，背后其实想要的是更快到达目的地。

   在这门课的语境里：用户说"我需要一个能提醒我作业的应用"，背后可能是"我需要在不依赖别人的情况下完成所有该完成的事"。一个是功能需求，一个是根本需求。

   做好的应用需要解决根本需求，而不只是满足表面功能。

2. **访谈设计：用"为什么"追问（15分钟）**

   学生为自己的项目设计一套5问访谈，包含：
   - 2个行为问题（你现在怎么做XXX？）
   - 2个追问（你刚才说XXX，为什么这样？那一步最麻烦的是哪里？）
   - 1个"如果不需要考虑技术和价格，你最理想的情况是什么？"

3. **角色扮演访谈（10分钟）**

   两人一组，互相访谈。"用户"要努力说真话，不要迎合"设计者"。

   关键练习：每当用户说了一个原因，"设计者"要追问一次"为什么"，不能满足于第一个答案。

4. **需求分析摘要（5分钟写，5分钟分享）**

   学生用以下格式写"需求分析摘要"（100字以内）：

   ```
   我访谈了___位用户。
   他们表达的需求是：_______________________
   但他们真正的根本需求是：_______________________
   我的应用要解决的，是：_______________________
   ```

**🎨 预期学生产出**

- 5问访谈记录（至少2次真实访谈）
- 一份"需求分析摘要"
- 明确写出"表面需求"和"根本需求"的区别

**📊 评估标准**

| 维度 | 未达到 | 进步中 | 已掌握 |
|------|--------|--------|--------|
| **追问深度** | 问完5个问题就停了，没有追问 | 有1-2次追问，但追问后不再深入 | 至少一次追到了用户的根本动机，不只是行为描述 |
| **区分需求层次** | 摘要中"表面需求"和"根本需求"基本相同 | 有区分，但区别不明显 | 清楚写出两者的不同，以及这个区别对项目的影响 |
| **样本质量** | 只访谈了1人，或访谈对象都是同班同学 | 访谈了2-3人，有不同来源 | 访谈了3人以上，包含不同角色（同龄人、家长、或其他相关人） |

**🌐 离线备用活动**

纸笔访谈。如果无法当场找到真实用户，老师扮演三种不同的"用户角色"，学生轮流向三个角色提问，老师根据不同角色特点给出真实反应（例如：一个非常忙的家长、一个不喜欢用手机的老师、一个记性不好的同学）。

## 课时三：设计决策与项目规划

### 小学版（6-10岁）

**🎯 教学目标**

学生能把调研发现的需求，转化成一个具体的项目规划：要做什么、给谁用、分几步做、什么时候完成。

**⏱ 时间安排**（50分钟）
- 导入（10分钟）：从需求到方案的翻译练习
- 主活动（30分钟）：填写独立项目规划模板
- 互评（10分钟）：同伴检查规划合理性

**📦 所需材料**
- 独立项目规划模板（每人一份，见本课程附录）
- 笔
- 上节课的访谈记录表

**📋 活动流程**

1. **从需求到方案（10分钟）**

   老师在黑板上写：
   - 用户的问题：每次放学不知道带了什么回家，经常忘东西
   - 解决方案A：做一个打印版清单，每天贴在书包上
   - 解决方案B：做一个网页，输入"今天要带什么"，可以打钩
   - 解决方案C：做一个AI助手，每天放学前提醒你检查书包

   问学生：这三个方案哪个最好？为什么？有没有哪个根本不需要用AI？

   关键引导：好的方案不一定是技术最复杂的，是最能解决问题的。

2. **填写独立项目规划模板（30分钟）**

   学生独立填写项目规划模板（见本文档附录，第"独立项目规划模板"节）。

   老师巡视，引导以下卡点：
   - "你说要解决XXX问题，你的应用具体怎么解决？用户在应用里能做什么？"
   - "你说两周做完，你知道有几步要做吗？第一步是什么？"

3. **同伴检查规划（10分钟）**

   两人交换规划，问对方三个问题：
   - "这个应用是给谁用的？你的规划里有体现吗？"
   - "这个应用能解决你说的问题吗？为什么？"
   - "你觉得他这个计划里，最可能卡住的步骤是哪一步？"

**🎨 预期学生产出**

- 一份完整的独立项目规划（所有格子填完，步骤可执行）
- 同伴评审上的3条批注

**📊 评估标准**

| 维度 | 未达到 | 进步中 | 已掌握 |
|------|--------|--------|--------|
| **问题与方案对应** | 方案和问题没有关系，或方案太宽泛（"做一个有用的应用"） | 有联系，但怎么解决说不清楚 | 能具体说明"用户用这个应用怎么操作，就能解决他的问题" |
| **步骤可行性** | 步骤很笼统（"先做这个，再做那个"），无法执行 | 步骤有了，但有些步骤太大或太模糊 | 步骤可以直接照做，每步有具体的产出（"做完第一步，会有XXX可以给别人看"） |
| **独立完成** | 规划内容基本是老师或同伴帮写的 | 大部分自己写，有几处需要引导 | 全部自己填写，同伴评审时只是微调，不是重写 |

**🌐 离线备用活动**

完全纸笔工作，不需要设备。

---

#### 中学版（11-15岁）

**🎯 教学目标**

学生能完成一份包含核心功能决策、放弃功能说明、时间线的完整项目规划，并能向同学解释每个主要决策的理由。

**⏱ 时间安排**（50分钟）
- 导入（10分钟）：功能优先级练习
- 主活动（30分钟）：项目规划 + 功能裁剪
- 决策辩护（10分钟）：为规划里的每个主要决定辩护

**📋 活动流程**

1. **功能优先级：什么必须做，什么可以不做（10分钟）**

   老师出一个练习：假设你要做一个"班级借书系统"，下面是候选功能——

   - 展示书单
   - 借书/还书操作
   - 查看谁借了哪本书
   - 按类别搜索
   - 发邮件提醒还书
   - 书的评分和评论
   - 图书封面上传

   问学生：如果只能选3个，你选哪3个？为什么？放弃的4个功能，你的理由是什么？

   引导学生理解：功能取舍不是能力问题，是设计判断问题。能说清楚"为什么不做"的设计者，比"什么都想加"的设计者更成熟。

2. **独立填写完整项目规划（25分钟）**

   使用本文档附录中的"独立项目规划模板"（中学版）。

   重点要填的部分：
   - **核心功能**（必须做，3个以内）
   - **主动放弃的功能**（想做但不做，原因）
   - **时间线**（从今天到 Demo Day，每周目标）
   - **用户测试计划**（第几周测试，测试谁，测什么）

3. **决策辩护（10分钟）**

   两人一组，轮流扮演"设计评审官"（不是攻击，是帮对方想清楚）。

   评审官只问两类问题：
   - "你为什么做这个功能，而不是XXX功能？"
   - "你为什么放弃了XXX，用户不需要吗？"

**🎨 预期学生产出**

- 完整的项目规划（包含核心功能、放弃功能、时间线）
- 同伴评审记录（至少2个被问到的决策问题，以及自己的回答）

**📊 评估标准**

| 维度 | 未达到 | 进步中 | 已掌握 |
|------|--------|--------|--------|
| **功能取舍** | 想做的功能全部列进去，没有说明优先级 | 列出了核心功能，但没有说明为什么其他不做 | 核心功能少而精，放弃的功能都有说明原因（不是"太难做"，而是"对用户价值不大"） |
| **时间线可行性** | 时间线要么太空（"第一周做完所有事"），要么没有时间线 | 有时间线，但分布不均匀 | 时间线合理，每周有具体目标，有用户测试节点 |
| **决策说明能力** | 被问到"为什么"时说不清楚，只能说"我觉得" | 能给出理由，但比较模糊（"因为用户可能需要"） | 能清楚说明决策理由，包括考虑过的替代方案和放弃它的原因 |

**🌐 离线备用活动**

全纸笔工作。额外挑战：在规划旁边，画出"如果某个步骤出了问题，我的备选方案是什么"，培养预案思维。

## 课时四：同伴评审与迭代

### 小学版（6-10岁）

**🎯 教学目标**

学生能按照同伴评审协议，给出具体有用的反馈（不只是"很好"），并能接受反馈、决定哪些要改。

**⏱ 时间安排**（50分钟）
- 导入（10分钟）：什么是"有用的反馈"
- 主活动（30分钟）：大带小同伴评审
- 决策与记录（10分钟）：收到反馈后我要改什么

**📦 所需材料**
- 同伴评审表（见本文档"同伴评审协议"节）
- 学生的项目（当前进度，能演示的版本）
- 笔

**📋 活动流程**

1. **什么是"有用的反馈"（10分钟）**

   老师在黑板写两列例子：

   | 没用的反馈 | 有用的反馈 |
   |-----------|-----------|
   | "做得很棒！" | "这个按钮我第一次看不懂是干嘛的，因为上面只有一个图标没有文字" |
   | "感觉不太好用" | "我找了三次才找到'添加'在哪里，它现在在右下角，我以为在上面" |
   | "加一个功能会更好" | "如果可以看到今天哪些人借了书就更方便了，因为我想等他还了再借" |

   总结：有用的反馈说的是具体的操作、具体的困惑、具体的理由。

2. **配对与评审（25分钟）**

   按照"大带小"原则配对（中学生 + 小学生，或高年级小学生 + 低年级小学生）。

   流程：
   - 第一步（5分钟）：设计者演示自己的项目，不解释，让评审者自己用
   - 第二步（10分钟）：评审者填写同伴评审表
   - 第三步（5分钟）：评审者口头说出最重要的一条反馈
   - 第四步（5分钟）：交换角色

   老师提示大学生：你不是来夸他的，也不是来挑剔的，是来帮他想清楚用户会遇到什么困难。

3. **决策：我要改什么（10分钟）**

   每人看完自己收到的反馈，在反馈表上标注：
   - ✅ 这条我要改（写一句："我要把XXX改成XXX"）
   - ❌ 这条我不改（写一句理由）

   老师要让学生明白：不改某条反馈不是错的，有理由地不改才重要。

**🎨 预期学生产出**

- 1份给对方写的同伴评审表（含至少3条具体反馈）
- 自己收到的反馈表，上面有标注（要改/不改+理由）
- 口头说出一句："我收到的最有用的反馈是……"

**📊 评估标准**

| 维度 | 未达到 | 进步中 | 已掌握 |
|------|--------|--------|--------|
| **给出的反馈质量** | 反馈都是"很好"或"不好"，没有具体细节 | 有一两条具体反馈，其余还是模糊 | 至少3条反馈都有具体的操作场景，说清楚"在哪里、做什么时、发生了什么" |
| **接受反馈的态度** | 听到反馈就说"不对，你没理解我的意思" | 听完了，但没有认真考虑，全部标"不改" | 认真区分，有的接受，有的不接受，不接受的给出了合理理由 |
| **迭代决策** | 说不出要改什么 | 知道要改，但说不清楚改成什么样 | 能说出"我要把XXX改成XXX，因为XXX" |

**🌐 离线备用活动**

本活动不需要网络。如果连电脑都没有，可以用纸质原型代替实际应用进行评审——学生把界面画在纸上，评审者"假装操作"，同样能产生真实反馈。

---

#### 中学版（11-15岁）

**🎯 教学目标**

学生能主导一次结构化同伴评审，既能给出产品层面（功能逻辑）和体验层面（使用感受）的双维度反馈，也能对收到的反馈做出有理由的取舍决策。

**⏱ 时间安排**（50分钟）
- 导入（10分钟）：产品评审和使用评审的区别
- 主活动（30分钟）：双维度评审实践
- 迭代规划（10分钟）：基于反馈，更新项目规划

**📋 活动流程**

1. **两种视角的评审（10分钟）**

   老师解释：
   - **产品视角**：这个应用逻辑上有没有漏洞？能不能解决它声称要解决的问题？
   - **体验视角**：用起来顺不顺？找东西找得到吗？有没有让人困惑的地方？

   一个好的同伴评审，要同时从这两个角度看。

2. **双维度评审（30分钟）**

   两人配对，每次评审15分钟：

   产品视角（7分钟）：
   - "你说这个应用解决的是XXX问题，我来检查它真的能解决吗"
   - 评审者扮演"挑战者"：找出应用无法处理的情况（如果用户做了X，这个应用能处理吗？）

   体验视角（8分钟）：
   - 评审者实际使用应用，完成一个具体任务，不接受任何提示
   - 设计者观察，记录评审者卡在哪里

3. **基于反馈更新规划（10分钟）**

   学生在项目规划里加一栏："第X轮评审收到的主要反馈 + 我的处理决策（改/不改+理由）"

**🎨 预期学生产出**

- 双维度评审记录（产品视角2-3条，体验视角2-3条）
- 更新后的项目规划（加入评审反馈处理记录）

**📊 评估标准**

| 维度 | 未达到 | 进步中 | 已掌握 |
|------|--------|--------|--------|
| **产品视角深度** | 没有从逻辑角度检查，只看了表面功能 | 提出了一个逻辑漏洞，但不够深入 | 找出至少一个"应用解决不了的边缘情况"，并说明为什么 |
| **体验视角精准** | 体验反馈是主观感受（"感觉不好"），没有具体行为描述 | 有具体行为，但没有说明为什么困惑 | 能描述"我在哪个步骤停了下来，因为我以为XXX，但实际是XXX" |
| **反馈取舍质量** | 全收或全不收，没有判断 | 有取舍，但理由不清楚 | 每个取舍决策都有明确理由，体现了对用户价值的判断 |

**🌐 离线备用活动**

纸质原型评审。如果项目还没有可运行版本，用纸质手绘界面代替，同样走完双维度评审流程。

## 问题发现工作坊：完整引导问题清单

以下问题供教师在课时一及项目初期使用。每组问题对应一个思维层次，从表面往深处引导。

### 第一层：观察生活

1. 今天早上从起床到到学校，你做过哪件事觉得"这个好麻烦"？
2. 你有没有见过家里的大人在做一件重复的事，你觉得"要是有工具帮他们做就好了"？
3. 你上周有没有忘记过什么东西？是什么？为什么忘了？
4. 你们班有没有什么事，总是会有人搞错或者搞乱的？

### 第二层：挖掘频率和影响

5. 你刚才说的那个麻烦，一周发生几次？
6. 因为这个问题，你或者其他人浪费了多少时间？
7. 有没有人因为这个问题而感到沮丧或不开心？是什么感觉？
8. 如果这个问题被解决了，会有什么不一样？谁受益最大？

### 第三层：检验是否可以用 AI 工具解决

9. 这个问题是信息问题（不知道某件事）、协调问题（要和其他人对齐），还是重复工作问题（每次都要做一遍的事）？
10. 这个问题，如果有一个工具能帮你的话，这个工具需要做什么？
11. 现在有没有任何东西在帮你部分解决这个问题？它哪里还做得不够好？

### 小学版引导问题（更简单、更具体）

1. 你最近一次因为"忘了"而出了问题是什么时候？
2. 你家里谁最忙？他们最忙的是做什么事？
3. 你们班有没有一件事，每次都要问老师或者同学才能知道的？
4. 你有没有想过"如果有个东西能帮我……就好了"？
5. 你最近一次等了很久的事是什么？等什么？为什么要等？

### 中学版额外深度问题

12. 这个问题有没有已经存在的解决方案？（比如某个应用、某种做法）它为什么没有被所有人用？
13. 解决这个问题的难点在哪里？是技术、是习惯、还是大家不在乎？
14. 如果你不做这个工具，三个月后这个问题会变好还是变坏？

## 同伴评审协议

### 大带小配对原则

- **配对规则**：中学生（11-15岁）评价小学生项目；高年级小学生也可以评价低年级小学生
- **配对时机**：课时四正式评审、顶石项目中期检查（大约到一半进度时）
- **轮次**：建议全程做2轮正式同伴评审

### 评审步骤清单（大学生跟着做）

1. 让小创作者**演示**他的项目，不要提示，自己用3分钟走一遍

5. 小创作者在评审表上标注"要改"或"不改"，写原因

## 顶石项目指南

### 什么是顶石项目

顶石项目是阶段四每位学生独立完成的一个真实应用。它不由老师指定题目，完全由学生自己发现问题、设计方案、构建完成，并在 Demo Day 上展示。

这是整个课程体系里，学生第一次真正意义上"为自己的想法负全责"的时刻。

### 项目标准

顶石项目的评估从以下四个维度衡量：

| 维度 | 说明 |
|------|------|
| **创意性** | 问题是学生主动发现的，不是照搬别人的，有自己的思考 |
| **实用性** | 应用解决了真实的问题，真实用户试用后有正面反馈 |
| **完成度** | 核心功能可以运行，用户可以走完完整的使用流程 |
| **表达能力** | 在 Demo Day 上能清晰说明：问题是什么、给谁解决、为什么这样设计 |

### 顶石项目评估量表

| 维度 | 未达到 | 进步中 | 已掌握 |
|------|--------|--------|--------|
| **创意性** | 问题来自老师或照搬别人，没有自己的思考 | 有自己的想法，但问题来源比较随意 | 问题来自主动观察，访谈验证了问题真实存在，解决方案有自己的设计判断 |
| **实用性** | 没有真实用户测试，或用户反馈是负面的 | 有用户测试，反馈一般，但没有根据反馈改进 | 至少两轮用户测试，每轮都有改进，至少一位用户说"我真的想用这个" |
| **完成度** | 核心功能无法运行，或只能在演示时看起来好用 | 一个以上核心功能可以运行，但有明显问题 | 所有规划的核心功能正常运行，用户能独立走完主要使用流程 |
| **表达能力** | 演示时说不清楚应用是什么，或只能介绍功能而说不出为什么 | 能说清楚应用的功能，但说不清楚设计决策 | 能在3分钟内说清楚：问题背景、给谁用、怎么解决、做了哪些重要决定、为什么这样决定 |

### 项目时间线建议

```
立项
（完成项目规划模板，问题已通过调研验证）
        ↓
第1周：搭出第一个核心功能的原型
（能运行，哪怕只有一个功能）
        ↓
第2周：完成第二、第三个核心功能
        ↓
第3周：第一轮用户测试
（找2-3位真实用户，收集反馈，记录在评审表上）
        ↓
第4周：根据第一轮反馈修改
（核心改动，不超过2-3处）
        ↓
第5周：第二轮用户测试
（可以找不同的用户，验证改进是否有效）
        ↓
第6周：打磨与准备 Demo Day
（修复发现的小问题，准备展示内容）
        ↓
Demo Day 展示
```

**注意**：时间线是参考，不是硬性要求。有些学生的项目可能更简单，4周就能完成两轮测试；有些更复杂的项目可能需要8周。老师根据学生情况灵活调整。

### 顶石项目检查点

每到一个关键节点，老师和学生做一次5分钟的检查对话：

**立项检查（第1课结束）**
- "你的问题是真实存在的吗？你怎么知道？"
- "你规划的核心功能能解决这个问题吗？"

**中期检查（大约到一半进度）**
- "你现在有可以给人看的东西吗？给我演示一下。"
- "你原来的计划哪里变了？为什么变？"

**测试前检查（第一轮用户测试前）**
- "你准备让用户做什么操作？"
- "你最担心他们在哪里卡住？"

**Demo Day 前检查（展示前1-2天）**
- "你的3分钟展示内容是什么？给我练一遍。"
- "家长问你'为什么这样设计'，你怎么回答？"

## Demo Day 完整策划方案

### 什么是 Demo Day

Demo Day 是本妙坊未来成长中心的年度展示日。所有完成阶段四顶石项目的学生，在这一天向家长、同学和老师公开展示自己的作品。

Demo Day 不是竞赛。没有排名，没有奖状，只有展示。每一件作品都代表了这个学生走过的真实历程，都值得被看见。

### 展示格式

每位学生：
- **3分钟演示**：介绍项目、现场展示应用、说明设计决策
- **2分钟问答**：回答观众（家长、同学）的提问

### 3分钟演示结构

```
第1分钟：问题背景
"我发现了一个问题——[描述问题，说明谁遇到、多久遇到一次]
我做了一个应用来解决这个问题，它叫[名称]。"

第2分钟：现场演示
[打开应用，走一遍核心操作流程]
"用户打开后，先看到的是……
然后他们可以……这样就解决了[问题]。"

第3分钟：设计决策
"我做了一个重要的决定——[某个功能的取舍或设计选择]
我选择这样做是因为……
我的用户试用后说……[引用一条真实反馈]"
```

### 2分钟问答

观众可以问任何问题。学生提前准备以下几类常见问题：

- "为什么你选了这个题目？"
- "你最难做的部分是什么？"
- "用户有没有说希望你再加什么功能？你做了吗？"
- "你用什么工具做的？"（标准回答：ClaudeCode，它帮我把我的想法变成了一个真正能运行的应用）
- "如果给你更多时间，你会继续改进什么？"

### 布场建议

**教室布置**

将教室按"展台"方式布置：

- 把桌子排成一排或U形，每个学生占一张桌子作为自己的展台
- 展台上放：笔记本电脑（已打开应用）、一张手写的"项目说明卡"（项目名称、解决的问题、一句话简介）
- 可在展台旁贴上用户反馈记录（手写的测试笔记）
- 教室正前方留出一个"主台"供整体介绍和集体问答用

**物料准备清单**

- [ ] 所有学生的应用已在设备上预先打开，确认可以运行
- [ ] 每位学生准备一张项目说明卡（手写，放在展台上）
- [ ] 家长参观路线（进门→每个展台→主台）已规划
- [ ] 备用电源/充电器就位
- [ ] 如果有网络要求，提前确认展示日网络可用

**座位安排**

- 家长参观可以流动，不用坐在固定位置
- 整体展示环节，学生坐前排，家长坐后排
- 每个学生至少有一位家长在展台旁陪同看演示

### 主持词提示

**开场（由教师或年长学生主持）**

> "欢迎来到本妙坊 Demo Day。今天大家看到的，是我们学生花了[X]个月做出来的真实作品——每一件都是从发现一个真实问题开始，经过了用户测试和修改，才站在这里展示。
>
> 今天没有排名，没有评分。每位同学的作品都代表了他们自己的探索历程。请大家自由参观每位同学的展台，欢迎提问——学生们已经准备好了回答你们的问题。"

**单个学生展示前**

> "[学生姓名]，告诉我们你发现了什么问题，你怎么解决了它。"

**整体结束语**

> "谢谢每一位同学今天的展示。你们学会了一件很重要的事：发现生活里的问题，然后用自己的方式去解决它。这件事不会随着今天结束——它会伴随你们很久。"

### 给家长的说明

在 Demo Day 开始前，可以发给家长一张简短说明：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
今天您的孩子会展示的内容：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

一个由他/她独立完成的应用程序。

他/她会告诉您：
• 发现了什么生活中的真实问题
• 这个应用是怎么解决这个问题的
• 做的过程中做了哪些重要决定，为什么

您可以问他/她：
• "你为什么选这个题目？"
• "做这个最难的部分是什么？"
• "有人试用过你的应用吗？他们怎么说？"

请注意：
• 今天没有排名或评分——每件作品都有它的价值
• 您看到的不是"完美的作品"，而是"经过真实探索的作品"
• 鼓励是最好的反馈，问题也是好的反馈

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 关于无排名的说明

Demo Day 没有排名，这不是因为要回避竞争，而是因为这里的作品没有可比性：

- 一个6岁学生解决了弟弟妹妹找不到睡前故事的问题，和一个14岁学生做了一个班级协调系统，两者之间没有"谁更好"，只有"谁在自己的能力边界上走得更远"。
- 我们想让每个学生在 Demo Day 上感受的，是"我做出来了"而不是"我比别人做得好"。前者让人继续创造，后者只让人焦虑。
- 每个作品都经过了真实的用户测试。能让真实用户觉得有用的作品，就是好作品。

## "天才通道"进阶活动

### 识别标准

"天才通道"不是给"最聪明"的学生的，是给**提前达到 L4 出口条件**的学生的。

具体可观测的标准：

- 学生在顶石项目规划后2周内就完成了所有核心功能，还有大量时间剩余
- 学生的用户测试反馈已经非常正面，而且他们还在主动寻找更难解决的问题
- 学生开始主动帮助其他同学，而不是因为老师要求
- 学生问出了这类问题："我的应用可以被真实的陌生人用吗？怎么让更多人知道它？"
- 学生开始思考"如果我的应用收集了用户数据，我应该怎么处理这些数据"之类的伦理问题

### 进阶活动一：真实用户扩展测试

**适合对象**：顶石项目完成度高、已经过至少2轮内部测试的学生

**挑战本质**：把应用给完全不认识的人用，面对真实陌生人的反应。这比给同学或家人测试难得多——陌生人不会迁就你，不会假装好用。

**具体做法**：

1. 学生准备一段30秒的"介绍脚本"（不透露解决方案，只说"这是一个解决XXX问题的工具"）
2. 学生找到至少3位与学校无关联的真实用户（可以是社区里的人、家长的朋友等，需要成人陪同）
3. 用户不接受任何操作提示，独立尝试使用
4. 学生写一份"陌生用户测试报告"：他们卡在哪里？他们的使用方式和预期有多大差距？

**产出物**：陌生用户测试报告 + 根据报告决定是否修改的说明

**为什么这有实质挑战**：同班同学了解你的项目背景，会替你"脑补"很多。陌生人会把你应用里所有含糊不清的地方全部暴露出来。这一步是真正检验应用是否独立可用的时刻。

---

### 进阶活动二：跨学科项目设计

**适合对象**：有明显跨领域兴趣的学生（例如对历史、音乐、生物感兴趣）

**挑战本质**：用 AI 工具解决另一个学科里的真实学习问题，并与那个学科的老师合作评估成果。

**具体做法**：

1. 学生选一门感兴趣的学科，访谈该学科的老师："你在教学时，有没有发现学生普遍遇到什么困难？有没有哪个环节总是需要反复解释？"
2. 学生把这个访谈结果变成一个问题陈述（和顶石项目的格式相同）
3. 学生在2-3周内，做出一个解决这个学科问题的小工具
4. 学生让该学科老师试用，收集反馈，做一次迭代

**示例方向**：
- 围棋老师说"复盘时学生不知道怎么描述每一手的用意" → 做一个复盘记录工具，让学生为每一手棋写一句"这手棋的目的是……"
- 数学老师说"学生不知道自己哪类题型错得最多" → 做一个错题分析工具，按题型记录和统计
- 语文老师说"学生写作文开头总是很难" → 做一个"开头生成器"，输入作文题目和关键词，给出几个不同风格的开头供选择

**产出物**：可被该学科老师实际使用的小工具 + 学科老师的使用反馈 + 学生的反思（跨学科做项目和做自己选题的项目，有什么不同）

---

### 进阶活动三：指导低年级同学

**适合对象**：在阶段四表现出明显引导能力的学生，同时自己的顶石项目已经基本完成

**挑战本质**：把自己学会的东西教给别人，是最深度的学习方式。教别人时会发现自己哪些东西其实没有真正理解，以及帮助别人需要一套完全不同的技能。

**具体做法**：

1. 学生选择一位阶段一或阶段二的低年级同学，作为"项目指导伙伴"
2. 每周2次，每次30分钟，学生陪同低年级同学做项目
3. 规则：不能替低年级同学做，只能问问题、提供方向、帮他们思考清楚
4. 学生需要写一份"指导记录"：今天他卡在哪里？我用了什么方式帮他想清楚？有没有用？

**关键约束**：低年级同学的作品必须是自己想的、自己做的，进阶活动学生只是"思考伙伴"，不是"操作者"。

**产出物**：5次指导记录 + 一份"我学到了什么关于教学的事"的反思（至少200字）

**为什么这有实质挑战**：教人很难。你自己会的东西，不代表你能解释清楚。找到对方卡在哪里、用对方能理解的方式引导，是一项独立的技能。很多进阶活动学生在做完这个后说："这比做我自己的项目难多了。"

---

### 天才通道的非正式结束

没有结业仪式，没有证书。当指导老师观察到学生开始"自发地帮助他人、自发地提出更大的问题"，这就是进阶活动完成最好的证明。

## 阶段四结束后的进阶路径

### 离开本中心后，怎么继续成长

完成阶段四的学生，已经具备了自主学习、独立创造的基础能力。离开学校之后，这些能力怎么继续往前走？

**路径一：继续解决真实问题**

每隔一段时间（建议每1-2个月），重新走一遍"发现问题→验证→构建→测试"的流程，解决一个自己遇到的新问题。不需要加入任何课程，只需要保持这个习惯。

你已经有了全套工具（ClaudeCode 或类似工具）和全套思维框架，剩下的就是练习次数。

**路径二：把应用给真实用户用**

如果顶石项目质量好，可以考虑：
- 分享给更多人用（从学校扩展到家人朋友所在的社区）
- 根据更多用户的反馈继续迭代
- 把应用分享到班级群或社区平台，看有没有陌生人主动来用

**路径三：探索更大的问题**

随着年龄增长，可以尝试解决规模更大的问题：
- 不只给自己班的同学用，而是给全校用
- 不只解决个人问题，而是解决一个群体共有的问题
- 和其他感兴趣的同龄人合作，分工完成更复杂的应用

**路径四：把自己的创造过程记录下来**

好的学习，不只在于做了什么，还在于能不能把做的过程说清楚。

建议：从现在开始，每完成一个项目，写一篇简短的"项目日记"（200-500字）：
- 我发现了什么问题
- 我怎么想的
- 遇到了什么困难，怎么解决的
- 如果再做一次，我会改变什么

几年之后，这些记录会变成最清晰的成长证明。

## 附录：教师使用参考

### 阶段四常见卡点与应对

| 学生常见问题 | 原因 | 应对方法 |
|------------|------|---------|
| "我找不到什么好问题" | 还在等"好问题"自动出现，没有主动观察 | 引导用问题地图工作坊，把观察范围从"我一天"扩大到"我周围的人一天" |
| "我的问题太小了，不值得做应用" | 对"值得"的标准错误（以为要解决世界级问题） | 提醒：能帮一个真实的人省时省力，就是值得。学校迷路导航、食谱搜索、作业确认，都是真实有价值的问题 |
| "调研了但不知道用调研结果做什么" | 访谈结果没有和项目规划连接起来 | 帮学生做"翻译"：用户说了什么 → 这意味着他需要什么功能 → 我的应用里有没有这个功能 |
| "做到一半不想做了" | 发现实际做起来比想象难，且不知道下一步 | 回到项目规划，问"你现在在时间线的哪个位置？下一步是什么？做出那一步需要做的最小的事是什么？" |
| "用户说好，但我觉得他是在安慰我" | 直觉是对的，但不知道怎么验证 | 引导用"行为测试"：不问用户"你觉得好不好"，而是"你愿意让你的朋友也用吗？你愿意明天再来用吗？" |
| "Demo Day 我不想讲" | 展示焦虑 | 先做私下预演：只给老师讲，然后加上一位同学，再加上两位。逐步扩大"观众数量" |

### 阶段四评估注意事项

- 不评估代码质量（老师不需要看代码）
- 不以功能多少判断好坏（一个功能做得透彻，好过三个功能做得浅）
- 评估重点：**学生能不能清楚解释他的设计决策**，这是 L4 最核心的能力
- 最重要的评估时机：Demo Day 问答环节，家长和同学的真实提问，是最客观的评估场

### Demo Day 前的教师准备

**提前2周**
- [ ] 确认每位学生的项目都有可运行版本
- [ ] 给每位学生安排一次5分钟的预演（只对老师讲）
- [ ] 通知家长 Demo Day 时间和参与方式

**提前1周**
- [ ] 全班集体预演（每人3分钟展示 + 2分钟同学问答）
- [ ] 讨论"被问到的最难问题是什么，怎么回答"
- [ ] 确认布场所需物资（电源、桌子排列、说明卡打印）

**Demo Day 当天**
- [ ] 提前30分钟布场，确认所有设备可以运行
- [ ] 门口放家长参观说明卡
- [ ] 每个展台摆好项目说明卡和访谈记录
- [ ] 老师在会场自由走动，不需要站在特定位置
