## Feynman Step 1：先用最简单的话解释
## Feynman Step 1: Explain It Simply

> 🎯 目标：为什么同样用 Vibe Coding，有人做出了真实产品，有人却只做出了一堆烂摊子？

**简单解释：**

工具人人都能用，但 **Vibe Coding 做得好不好，90% 取决于你怎么跟 AI 说话**。

这包括三件事：

1. **提示词（Prompt）** — 你一次说多少、说多清楚、怎么组织
2. **工作流（Workflow）** — 你先做什么、后做什么、什么时候让 AI 干、什么时候自己决定
3. **文档（Documentation）** — 你告诉 AI 这个项目是什么、有什么规则、不能干什么

**类比：**

Vibe Coding 像是**指挥一个刚入职的实习生**：

- 实习生（AI）很聪明、学得快，但**对你的项目一无所知**
- 你告诉他"帮我做个登录页面" → 他做出来的是他想象中的登录页面，可能完全不符合你的需求
- 你告诉他"这是我们的设计规范，用户名用邮箱，密码最少 8 位，按钮颜色用 #3B82F6，错误提示在输入框下方红色显示，做一个登录页面" → 结果会好很多
- 你还给他一本**项目手册**（AGENTS.md/CLAUDE.md），告诉他所有背景知识 → 他能独立工作，越来越顺

## Feynman Step 3：填漏洞
## Feynman Step 3: Fill the Gaps

### 漏洞 1：怎么写才叫"好的提示词"？

**原则一：把结果描述清楚，而不是步骤**

| ❌ 差 | ✅ 好 |
|------|------|
| "帮我做一个网页" | "做一个响应式的商品展示页，包含：顶部导航（Logo + 搜索框 + 购物车图标）、商品网格（每行 4 个卡片，卡片含图片/名称/价格/加购按钮），深色主题，使用 Tailwind CSS" |
| "修一下这个" | "第 23 行有报错：`TypeError: map is not a function`。原因是 `products` 变量有时候是 `undefined`，请加默认值 `[]` 并加一个 loading 状态" |
| "让它更好看" | "卡片加圆角（12px）、阴影（shadow-md），鼠标悬停时卡片轻微上移（translateY -4px），过渡动画 0.2s" |

**原则二：一次只做一件事**

```
❌ 一次做太多：
"帮我做登录、注册、忘记密码、个人中心、商品列表、购物车、结算页面"

✅ 分步来：
Round 1: "先做登录页面，包含邮箱+密码输入框、登录按钮，有表单验证"
Round 2: "登录成功后跳转到首页，失败时在密码框下方显示红色错误提示"
Round 3: "现在做注册页面，和登录页面风格一致，额外需要确认密码字段"
```

**原则三：给上下文（Context）**

AI 不知道你的项目是什么，每次对话都要简单说明背景：

```
好的开场白模板：
"我在做一个[项目类型]，技术栈是[前端/后端/数据库]，
目前已有[已完成的部分]，现在需要[这次的具体任务]。
要求：[具体要求1，要求2，要求3]"

示例：
"我在做一个在线记账 App，前端用 React + Tailwind，
后端 Node.js + Express，数据库 SQLite。目前登录功能已完成，
现在需要做'添加账单'功能：弹出表单，包含金额（数字输入）、
类别（下拉选择：餐饮/交通/娱乐/其他）、备注（可选），
提交后更新账单列表，不刷新页面。"
```

**原则四：学会迭代，不要期待一次完美**

```
第一次提示 → 看结果 → 描述哪里不对 → AI 修改 → 再看 → 再描述

每次迭代聚焦一个问题：
"按钮颜色不对，改成绿色"
"表单验证没有触发，提交空表单应该报错"
"手机端布局乱了，修一下响应式"
```

### 漏洞 2：真实项目的工作流是什么样的？

**专业 Vibe Coding 工作流（5 阶段）：**

```
阶段 1：规划（Planning）
─────────────────────────────────────────
你的工作：
  - 把大目标拆成小模块（登录、首页、商品页、购物车……）
  - 每个模块再拆成具体功能（登录 = 表单 + 验证 + API + 错误处理）
  - 写下来，形成任务清单

AI 的工作：
  - 用 Claude Task Master / vibe-kanban 让 AI 帮你拆解
  - 或者直接问 AI："帮我把这个项目拆分成合理的开发任务"

⚠️ 错误做法：直接跳到执行，让 AI "帮我做整个项目"

阶段 2：搭架子（Scaffolding）
─────────────────────────────────────────
你的工作：
  - 让 AI 生成项目骨架（文件结构、基础配置、路由）
  - 确认技术栈和目录结构符合预期

AI 的工作：
  - 生成初始代码结构
  - 安装依赖，配置基础环境

阶段 3：功能开发（Feature Development）
─────────────────────────────────────────
规律：一次一个功能，做完再做下一个

每个功能的小循环：
  提示 → 生成 → 测试 → 发现问题 → 描述问题 → 修复 → 确认OK → 下一个功能

⚠️ 重要：功能之间要经常提交（git commit），这样 AI 搞坏了可以回滚

阶段 4：收尾（Polish）
─────────────────────────────────────────
你的工作：
  - 检查整体一致性（样式、交互、错误处理）
  - 测试各种边界情况

AI 的工作：
  - 帮你找 bug、优化 UI
  - 补充错误处理、加 loading 状态

阶段 5：上线（Deployment）
─────────────────────────────────────────
浏览器工具：通常一键部署
CLI/IDE 工具：告诉 AI "帮我部署到 Vercel / Netlify"，它会引导你
```

**关键习惯：频繁 Git 提交**

```bash
# 每完成一个功能就提交一次
git add .
git commit -m "完成登录功能"

# 好处：AI 搞坏代码时，可以恢复
git checkout -- .   # 丢弃 AI 的修改，回到上一个提交
```

### 漏洞 3：AGENTS.md 和 CLAUDE.md 写什么？

**这类文件的本质：** 给 AI 的"项目说明书"——每次它进入你的项目，都会先读这个文件，了解背景和规则。

**一个实用的 AGENTS.md 模板：**

```markdown
# 项目说明 / Project Overview

## 这是什么项目
[一段话描述你的项目：做什么的、给谁用的]

## 技术栈
- 前端：React 18 + TypeScript + Tailwind CSS
- 后端：Node.js + Express
- 数据库：PostgreSQL
- 部署：Vercel (前端) + Railway (后端)

## 项目结构
src/
  components/   # UI 组件
  pages/        # 页面
  api/          # API 调用
  utils/        # 工具函数

## 编码规范
- 用 TypeScript，不允许 any 类型
- 组件用函数式写法（不用 class）
- CSS 只用 Tailwind，不写内联样式
- 文件命名：组件用 PascalCase，其他用 camelCase

## 禁止事项
- 不要修改 /src/auth/ 下的文件（正在另一个分支开发）
- 不要安装新的 npm 包，除非我明确同意
- 不要删除任何现有文件

## 当前任务背景
[可选：描述你现在在做什么阶段]
```

**Context7 的作用：**

AI 模型的训练数据有截止日期，可能不知道最新版本的 API。**Context7** 能实时把最新文档注入对话：

```
没有 Context7：
你："用 Next.js 15 的 App Router 做一个服务器组件"
AI：可能用了 Next.js 13 的旧写法，导致报错

有了 Context7（作为 MCP 接入）：
AI 自动获取 Next.js 最新文档 → 用正确的 API → 减少报错
```

### 漏洞 4：上下文窗口是什么，为什么 AI 会"忘事"？

**简单解释：**

AI 的"记忆"不是永久的，它有一个**上下文窗口（Context Window）**——类似于一张固定大小的便利贴。

- Claude 的上下文窗口大约是 20 万个 token（约 15 万个汉字，或者 10 万行代码）
- 对话越来越长，早期的内容会被"挤出"窗口，AI 就开始"忘记"
- 当 AI 忘记了之前的决定和代码结构，它就会开始做出不一致的选择

**解决方案：**

| 方法 | 怎么做 |
|------|-------|
| **定期重置对话** | 完成一个功能后，开新对话，在新对话里重新粘贴项目背景 |
| **写 AGENTS.md** | 让背景知识不依赖对话历史，每次新对话都能读到 |
| **分小任务** | 每次对话只做一个功能，不要在一个对话里做太多 |
| **用支持长上下文的工具** | Plandex 支持 200 万 token 上下文，专门处理大项目 |

## Feynman Step 4：精炼解释
## Feynman Step 4: Refined Explanation

**最终简单解释：**

Vibe Coding 做得好的关键是三件事：**好的提示词、好的工作流、好的文档**。

好的提示词要具体（描述结果，不描述步骤）、有上下文（告诉 AI 项目背景）、一次一件事（不要贪多）。好的工作流要分阶段（规划→搭架子→功能→收尾→上线）、频繁 commit（方便回滚）、每次对话专注一个任务。好的文档（AGENTS.md/CLAUDE.md）让 AI 始终了解你的项目规则，不会因为对话太长而"迷失"。

**精炼类比：**

做好 Vibe Coding 就像做好**远程协作**：

- 你是项目经理（PM），AI 是远程开发者
- 差的 PM：发一句"做个 App 出来"，然后等着
- 好的 PM：写清楚需求文档（提示词）、把任务拆成小块（工作流）、维护项目 Wiki（AGENTS.md）、每天同步进展（commit）

好的 PM 不需要懂所有技术细节，但**必须能清晰表达需求、分解任务、验收结果**。

**核心要点：**
1. 提示词质量 = 结果质量。具体、有上下文、一次一件事
2. 工作流是成败关键：规划→搭架子→功能循环→收尾→上线
3. 频繁 git commit 是保险丝，可以随时恢复 AI 搞坏的代码
4. AGENTS.md 让 AI 永远了解你的项目，解决"AI 忘事"问题

> **30 秒版本：**
> "跟 AI 说话要具体——告诉它背景、要什么结果、一次只做一件事。遇到大项目先规划再动手，每完成一个功能就 git commit 一次。写一个 AGENTS.md 文件告诉 AI 你的项目规则，这样每次新对话它都能快速上手。"

### 练习 3-B：完整工作流实战（约 60 分钟）

用**完整的 5 阶段工作流**做一个真实项目：

**要求：**
1. **先规划**：把功能拆成小任务，写到纸上

4. **记录提示词**：把你用的提示词都记下来，好的和坏的

### 练习 3-C：写你的第一个 AGENTS.md（约 20 分钟）

为你在 3-B 中做的项目写一个 AGENTS.md 文件，包含：

- 技术栈

- 至少 1 条"禁止事项"

### 练习 3-D：上下文窗口实验（约 15 分钟）

在同一个对话里，让 AI 做 5 件不同的事（可以是：改颜色、加功能、修 bug、加动画、改布局）。

在第 5 件事的时候，检查：
- AI 还记得你在第 1 件事里说的规则吗？
- 有没有出现前后矛盾的代码？
- 记录你观察到的"遗忘"现象
